## Виконай з кореня проєкту (~/IdeaProjects/PostExAgent/):jpackage \
  --type app-image \
  --input target \
  --name PostExAgent \
  --main-jar PostExAgent-1.0-SNAPSHOT.jar \
  --main-class com.agent.postexagent.Main \
  --runtime-image ~/tools/win-jre/jdk-17.0.11+9-jre \
  --dest dist/windows

буде створено: dist/windows/PostExAgent/PostExAgent.exe

## В Java существует два вида потоков 
- **пользовательские** которые мы создаем . Иногда их называют *основными* потоками выполнения кода
- потоки **демоны** потоки существующие в фоновом режиме например гарбач коллектор.


В Java существует несколько способов создания и использования потоков . Вот основные способы:

### 1. Реализация [[интерфейса ]] `Runnable` 

Вы можете создать поток, реализуя интерфейс `Runnable`. Это позволяет вашему классу выполнять задачи в отдельном потоке без необходимости наследования от класса `Thread`.

java

Copy code

`class MyRunnable implements Runnable {     @Override     public void run() {         System.out.println("Thread is running");     } }  public class Main {     public static void main(String[] args) {         MyRunnable myRunnable = new MyRunnable();         Thread thread = new Thread(myRunnable);         thread.start();     } }`

### 2. [[наследование]] от класса `Thread`

Вы можете создать поток, наследуя от класса `Thread`. Это позволяет вашему классу иметь собственный метод `run()`, который будет выполнен в отдельном потоке.

java

Copy code

`class MyThread extends Thread {     @Override     public void run() {         System.out.println("Thread is running");     } }  public class Main {     public static void main(String[] args) {         MyThread myThread = new MyThread();         myThread.start();     } }`

### 3. Использование лямбда-выражений

С введением лямбда-выражений в Java 8, создание потоков стало проще. Вы можете использовать лямбда-выражения для создания экземпляров `Runnable`.

java

Copy code

`public class Main {     public static void main(String[] args) {         Runnable runnable = () -> System.out.println("Thread is running");         Thread thread = new Thread(runnable);         thread.start();     } }`

### 4. Использование [[интерфейса]] `Callable` и `Future`

Если вам нужно, чтобы поток возвращал результат или бросал исключение, вы можете использовать [[интерфейсы]] `Callable` и `Future`.

java

Copy code

`import java.util.concurrent.Callable; import java.util.concurrent.ExecutionException; import java.util.concurrent.ExecutorService; import java.util.concurrent.Executors; import java.util.concurrent.Future;  public class Main {     public static void main(String[] args) {         ExecutorService executor = Executors.newSingleThreadExecutor();         Callable<String> callable = () -> {             return "Thread is running";         };          Future<String> future = executor.submit(callable);         try {             String result = future.get();             System.out.println(result);         } catch (InterruptedException | ExecutionException e) {             e.printStackTrace();         } finally {             executor.shutdown();         }     } }`

### 5. Использование ExecutorService

`ExecutorService` из пакета `java.util.concurrent` предоставляет более гибкие и мощные механизмы для управления потоками.

java

Copy code

`import java.util.concurrent.ExecutorService; import java.util.concurrent.Executors;  public class Main {     public static void main(String[] args) {         ExecutorService executor = Executors.newFixedThreadPool(3);          Runnable task1 = () -> System.out.println("Task 1");         Runnable task2 = () -> System.out.println("Task 2");         Runnable task3 = () -> System.out.println("Task 3");          executor.submit(task1);         executor.submit(task2);         executor.submit(task3);          executor.shutdown();     } }`

Каждый из этих способов имеет свои преимущества и может использоваться в зависимости от конкретной задачи.

@PutMapping("/all")  
public ResponseEntity<Book> updateBook(@RequestBody Book book) {  
  
    if (library.contains(book)) {  
  
        int index = library.indexOf(book);  
        library.set(index, book);  
        return new ResponseEntity<>(book, HttpStatus.OK);

        library.add(book);  
        return new ResponseEntity<>(book, HttpStatus.CREATED);

## обединения трех таблиц

Dependensy ingecten внедрение зависимости это техника которая позволяет передавать зависимости в объект из вне.через конструкторы, методы или свойства объектов

**Dependensy ingecten DI** это конкретный способ  передавать зависимости объекту из вне через конструктор.

**Dependensy inversion DI** 
зависимости высокоуровневых модулей не должны зависеть от низкоуровневых модулей то есть классы должны зависеть от интерфейсов или абстрактных классов а абстракции не зависят от их реализаций.
это реализация **D** принципа solid
(**dependency inversion principle** *Принцип инверсии зависимостей* - Если в приложении несколько уровней модулей, то верхние модули не должны зависеть от нижних)
 
**Dependensy inversion DI**  ***принцип*** как организовывать зависимости нашей системы
**Dependensy ingecten DI** ***конкретный способ*** реализации этого принципа

- **Репозитории** предоставляют информацию о запросах к базе данных и могут помочь в диагностике проблем с производительностью или ошибками в запросах.
- **Исключения и обработчики ошибок** помогают в отслеживании и устранении проблем, которые могут возникать в приложении.
- **Конфигурационные классы** и **утилиты** помогают в диагностике проблем конфигурации и выполнения.
для коня: 
применить логирование с умом. лишнего не надо, а только там где может быстрее всего понадобиться.

Сильні дієслова в німецькій мові – це ті, що змінюють кореневу голосну при відмінюванні (наприклад, у претериті чи партиципі II). Вони поділяються на кілька груп залежно від способу зміни кореневої голосної. Ось кілька прикладів сильних дієслів:

# Сильні дієслова в німецькій мові

Сильні дієслова утворюють форми минулого часу (Präteritum) і партиципу II (Partizip II) через зміну кореневої голосної. Вони мають власну систему відмінювання, яку важливо знати для правильного використання.

## Таблиця сильних дієслів за групами

### Група 1: e → a → o
| **Дієслово** | **Переклад** | **Präsens** | **Präteritum** | **Partizip II** | **Особливості** |
|--------------|--------------|--------------|----------------|-----------------|-----------------|
| geben        | давати       | ich **gebe** | ich **gab**    | ich habe **gegeben** | - |
| nehmen       | брати        | ich **nehme**| ich **nahm**   | ich habe **genommen**| - |
| essen        | їсти         | ich **esse** | ich **aß**     | ich habe **gegessen**| - |

### Група 2: i → a → u
| **Дієслово** | **Переклад** | **Präsens** | **Präteritum** | **Partizip II** | **Особливості** |
|--------------|--------------|--------------|----------------|-----------------|-----------------|
| singen       | співати      | ich **singe**| ich **sang**   | ich habe **gesungen**| - |
| springen     | стрибати     | ich **springe**| ich **sprang** | ich bin **gesprungen**| Використовується з **sein** |
| trinken      | пити         | ich **trinke**| ich **trank**  | ich habe **getrunken**| - |

### Група 3: e → a → e
| **Дієслово** | **Переклад** | **Präsens** | **Präteritum** | **Partizip II** | **Особливості** |
|--------------|--------------|--------------|----------------|-----------------|-----------------|
| lesen        | читати       | ich **lese** | ich **las**    | ich habe **gelesen**| - |
| sehen        | бачити       | ich **sehe** | ich **sah**    | ich habe **gesehen**| - |

### Група 4: a → u → a
| **Дієслово** | **Переклад** | **Präsens** | **Präteritum** | **Partizip II** | **Особливості** |
|--------------|--------------|--------------|----------------|-----------------|-----------------|
| fahren       | їхати        | ich **fahre**| ich **fuhr**   | ich bin **gefahren**| Використовується з **sein** |
| tragen       | нести        | ich **trage**| ich **trug**   | ich habe **getragen**| - |

### Група 5: ei → ie → ie
| **Дієслово** | **Переклад** | **Präsens** | **Präteritum** | **Partizip II** | **Особливості** |
|--------------|--------------|--------------|----------------|-----------------|-----------------|
| bleiben      | залишатися   | ich **bleibe**| ich **blieb** | ich bin **geblieben**| Використовується з **sein** |
| schreiben    | писати       | ich **schreibe**| ich **schrieb**| ich habe **geschrieben**| - |

### Інші приклади
| **Дієслово** | **Переклад** | **Präsens** | **Präteritum** | **Partizip II** | **Особливості** |
|--------------|--------------|--------------|----------------|-----------------|-----------------|
| sein         | бути         | ich **bin** | ich **war**    | ich bin **gewesen**| Нерегулярне |
| werden       | ставати      | ich **werde**| ich **wurde**  | ich bin **geworden**| Нерегулярне |
| kommen       | приходити    | ich **komme**| ich **kam**    | ich bin **gekommen**| Використовується з **sein** |

---

Вивчення сильних дієслів є важливим для розуміння граматики німецької мови, адже вони використовуються у багатьох повсякденних ситуаціях.


Сильних дієслів у німецькій мові доволі багато, і вони часто використовуються, тому їх важливо запам'ятовувати.

### принципы  методологического подхода
DRY - Don't repeat yourself - не повторяйся

SSOT - single source of truth - единый источник истины-
практика структурирования информацинонных моделей и схем данных *(все фрагменты данных обрабатываются редактируются в **одном**  месте)*


KISS - Keep it simple, stupid
       Keep it stupid simple- (принцип ВМС США) 
       *Простые системы будут работать лучше и надежнее сложных*


SOLID
- single responsibility principle (SRP) *Принцип единой ответственности* -- каждый метод должен выполнять только **одну** работу(действие)

- open-close principle - *Принцип открытости(закрытости)* - Модули должны быть ***открыты*** для **расширения** и ***закрыты*** для **модификации** 

- Liskov substitution principle (LSP)  *Принцип подстановки Барбара Лискофф* -  Методы которые используют ссылки на базовые классы должны иметь возможность использовать наследников базового класса (ссылки на дочерние классы) ничего не зная об их существовании


- interface segregation principle *Принцип разделения [[интерфейса]]* -  нельзя заставлять програмиста который будет работать с вашим кодом использовать ненужные методы. 


- dependency inversion principle *Принцип инверсии зависимостей* - Если в приложении несколько уровней модулей, то верхние модули не должны зависеть от нижних.

YAGNI создавать какую то реальную функциональность в коде требуется только тогда, когда она действительно нужна.

*NIX - системы (линукс, макОС) выполняется только одна операция.

Переносит изменения из рабочего каталога в раздел проиндексированных файлов. В этом разделе можно подготовить снимок состояния, прежде чем сделать коммит изменений в официальную историю.

С командой git checkout можно не только получать старые коммиты и прежние версии файлов, но и осуществлять навигацию по существующим веткам. В сочетании с базовыми командами Git она позволяет сосредоточиться на определенном направлении разработки.

[[HTTP]] протокол базируется на запросеот клиента серверу

**клиент**  тот кто может выполнить [[HTTP]] **[[запрос]]**  // В качестве **клиента** выступает **браузер** 
сервер программа которая может получить запрос и отправить **[[ответ]]** // В качестве **сервера** выступает некий  **веб-сервер** который умеет отвечать на запросы
в проэкте сервером выступает **джарник** который умеет отвечать на запросы со стороны браузера 
в этом случае на моем компютере поднимается **томкат** сервис и мой комп становится сервером
## запрос
первый рядок - информация о методе, адресе и версии [[HTTP]] протокола
{**метод**} {**адрес** } {  **НТТР и версия**}

после этого идет **заголовок** в формате:
**ключ** : **значение** 
если есть **тело** то оно печатается после **заголовков** и отделяется от них пустым рядком
GET /wiki/Main_Page HTTP
Host : en.wikipendia.org

POST /pages HTTP/1.1
Host : exsample.com
User-Agent : firefox/5.0( Linux; Debian 5.0.8)

## ответ
в первом рядке ответа идет информация об версии НТТР , код состояния и опись состояния
{версии НТТР } {код состояния } {опись состояния } 
HTTP/1.1 200 OK
Server : Apache
Content-Type : text/html; .................
Content- Length : 1234
body answer.(тело ответа)

## автоконфигурация

для автоматического связывания классов их помечают специальной аннотацией @**ComponentScan** при этом конфигурационный файл остается таккой же или тот же как в случае с **джава конфиг**
@ComponentScan(**basePackages = "de.telran.springcontainerconfigexample.beans**")

в аннотации **@ComponentScan** указывается конкретное место расположения бинов (**basePackages = "de.telran.springcontainerconfigexample.beans**")
@PropertySource("classpath:application.properties") указывает путь к проперти которые находятся в aplication.properties

в автоконфигурации джава сама ищет бины то есть классы которые помечаются  @**Component**   но эти классы нужно потом инициировать 
можно создать пустой конструктор и в нем передать поля класса например:
public class Country {  
    private String name;  
    private String code;  
    private String languale;  
  
    public Country() {  
        this.setName("France");  
        this.setCode("FR");  
        this.setLanguale("French");  
    }
**связывание бинов**
чтобы связывать классы используют три способа:
- связывание полей
**@Autowired**  
**private Country country**;

- связывание через сеттер 
производится с указанием бина **@Autowired** сеттерам , кроме того нужен пустой конструктор , например:

@Autowired  
public void setValidator(Validator validator) {  
    this.validator = validator;  
}  
  
@Autowired  
public void setCountry(Country country) {  
    this.country = country;  
}

- связывание через конструктор
производится с указанием бина **@Autowired** конструктору, например:

@Autowired  
public UseBeans(Validator validator, Country country) {  
    this.validator = validator;  
    this.country = country;  
}

Для передачи данных по сети необходимо установить и настроить протокол — пра-
вила, по которым два удаленных устройства будут обмениваться информацией.
## Адресация

Основным для Linux является ставший стандартом для Интернета протокол **TCP/IP**
**(Transmission Control Protocol / Internet Protocol,** протокол управления передачей 
протокол [[интернет]]) версии 4,
Для того чтобы сеть работала, нам необходимо как минимум настроить следующие
параметры:
- *указать адрес*
Вся адресация в сети происходит по **IP-адресу**
Для подключения по **локальной** сети адреса задаются **самостоятельно**
стоит ограничиться адресами вида **192.168.1.х**, где х — число от 1 до 254
номер компютера це останнє число - 001 або (77) 1 або інше. одиницч(1) тут ш є номер компютера.
- *установить маску подсети*
- 255.0.0.0 (маска /8)
255.255.0.0 (маска /16)
255.255.255.0 (маска /24)
255.255.255.128 (маска /25)
[[номера масок до 30]] 
## Информация о сетевых подключениях
Для получения информации о текущей настройке сетевых карт и протокола TCP/IP
необходимо выполнить команду ifconfig
## Изменение параметров сетевого подключения
 [[Аналіз мереж і управління ними]], 
Для получения информации об IP-адресах используется утилита ip те ж саме робить ifconfig(заст)
**ip addr**
Если выполнить эту команду с параметром **link**, можно увидеть состояние сетевых
интерфейсов:
**ip link**
Як додати новий IP-адрес до інтерфейсу:
**sudo ip addr add 192.168.1.100/24 dev eth0**
Після того, як ви призначили IP-адресу, можна перевірити підключення, використовуючи команду **ping**
**ping 192.168.1.1**

## Базовые настройки сети
 **hostname** С помощью команды **hostname** можно просмотреть имя компьютера (хоста)

**hostname server** команда устанавливает имя хоста в значение **server** Эта команда меняет имя хоста, но такое изменение **потеряет силу** после перезагрузки компьютера.
**hostnamectl** поменять имя хоста и сразу сохранить его в файле
**hostnamectl set-hostname srv.flenov.info**
**set-hostname** Ключ указывает на то, что мы хотим поменять имя компьютера на
**srv.flenov.info**.

- NEW - новый поток создан, обэявлен но **не** запущен

         - **teimwaiting** ожидает только определенное время, потом завершает работу
-  DEAD состояние после завершения или возникновения какой-то ошибки

если два потока работают с каким - тоодним объектом или хотят к нему обратиться

**volatile** ключевое слово используется в ситуации когда мы *хотим* объявить переменную, которая может быть *использована* несколькими потоками

оно гарантирует что любое изменение такой переменной будут видны другим потокам сразу же после изменения значения этой переменной
 ускоряется работа за счет того что не нужно блокировать доступ к данным другим потокам, так как они уже уведомлены об этом и могут оперировать этими данными

StringBuffer **потокобезопасный**

 @PostMapping("/all") void addBook(@RequestBody Book book) {

@PostMapping("/all")  
public ResponseEntity<Book> addBook(@RequestBody Book book) {

## автоконфигурация

для автоматического связывания классов их помечают специальной аннотацией @**ComponentScan** при этом конфигурационный файл остается таккой же или тот же как в случае с **джава конфиг**
@ComponentScan(**basePackages = "de.telran.springcontainerconfigexample.beans**")

в аннотации **@ComponentScan** указывается конкретное место расположения бинов (**basePackages = "de.telran.springcontainerconfigexample.beans**")
@PropertySource("classpath:application.properties") указывает путь к проперти которые находятся в aplication.properties

в автоконфигурации джава сама ищет бины то есть классы которые помечаются  @**Component**   но эти классы нужно потом инициировать 
можно создать пустой конструктор и в нем передать поля класса например:
public class Country {
